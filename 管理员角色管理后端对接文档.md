# 管理员角色管理后端对接文档

## 1. 功能背景

本次前端新增了“管理员角色管理”能力，目标是支持：

- 管理员查看用户列表
- `super-admin` 设置管理员、回收管理员
- `admin` 禁用用户
- 登录后根据用户角色动态展示“管理后台”入口
- 个人中心根据角色同步管理员状态
- 后续支持管理员数据审核、导入导出等后台功能

当前前端角色体系为：

- `super-admin`
- `admin`
- `member`

其中：

- `super-admin` 和 `admin` 都视为管理员
- `member` 为普通用户
- 前端管理员显隐判断基于 `role` 字段

---

## 2. 角色与权限规则

### 2.1 角色定义

| 角色值 | 含义 | 前端是否视为管理员 |
| --- | --- | --- |
| `super-admin` | 超级管理员 | 是 |
| `admin` | 普通管理员 | 是 |
| `member` | 普通用户 | 否 |

### 2.2 前端权限约束

前端当前按以下逻辑处理：

- `super-admin`
- 可查看全部用户
- 可将 `member` 设置为 `admin`
- 可将 `admin` 回收为 `member`
- 不允许操作自身超级管理员固定身份逻辑
- `admin`
- 不可操作 `super-admin`
- 可对非 `super-admin` 用户执行“禁用用户”
- `member`
- 不显示管理后台入口

### 2.3 前端管理员判断逻辑

前端使用如下逻辑判断是否为管理员：

```js
role === 'admin' || role === 'super-admin'
```

## 3. 接口清单

当前前端已接入的管理员相关接口定义文件：

### 3.1 获取管理员首页汇总

**接口**

- `GET /admin/dashboard`

**用途**

- 管理后台首页展示待审核、已审核、已驳回、最近动态

### 3.2 获取审核列表

**接口**

- `GET /admin/audit`

**用途**

- 管理员审核页展示志愿服务/荣誉获奖记录

### 3.3 提交审核操作

**接口**

- `POST /admin/audit`

**用途**

- 单条通过
- 单条驳回
- 批量通过
- 批量驳回
- 调整积分
- 调整荣誉级别

### 3.4 批量导入

**接口**

- `POST /admin/import`

**用途**

- 管理员上传 Excel 后发起导入

### 3.5 全量导出

**接口**

- `GET /admin/export`

**用途**

- 根据条件导出审核数据

### 3.6 获取用户列表

**接口**

- `GET /admin/users`

**用途**

- 管理员角色管理页获取用户列表

**前端使用场景**

- 角色筛选
- 状态筛选
- 搜索姓名/手机号
- 分页拉全量

### 3.7 设置/回收管理员

**接口**

- `POST /admin/users/role`

**用途**

- `super-admin` 将普通用户设为管理员
- `super-admin` 将管理员回收为普通用户

### 3.8 禁用用户

**接口**

- `POST /admin/users/disable`

**用途**

- `admin` 禁用普通用户或管理员账号
- 不允许禁用 `super-admin`

### 3.9 登录

**接口**

- `POST /auth/login`

**用途**

- 微信登录换取 `openid`、`token`、用户角色信息

### 3.10 提交实名认证信息

**接口**

- `POST /user/realname`

**用途**

- 首次登录时补充用户信息

### 3.11 获取用户资料

**接口**

- `GET /user/profile`

**用途**

- 个人中心展示资料、积分
- 同步角色信息
- 同步管理员状态

## 4. 请求与响应字段要求

### 4.1 `POST /auth/login`

#### 前端请求参数

```json
{
  "code": "微信登录 code"
}
```

#### 前端期望响应

```json
{
  "openid": "用户openid",
  "token": "登录token",
  "needBinding": true,
  "userInfo": {
    "realName": "张三",
    "phone": "13800000000",
    "role": "member"
  }
}
```

#### 字段说明

- `openid`：登录态标识
- `token`：后续接口鉴权
- `needBinding`：是否需要补填实名认证信息
- `userInfo.role`：角色判断关键字段

### 4.2 `POST /user/realname`

#### 前端请求参数

```json
{
  "realName": "张三",
  "phone": "13800000000",
  "openid": "用户openid"
}
```

#### 前端期望响应

```json
{
  "openid": "用户openid",
  "token": "登录token",
  "userInfo": {
    "realName": "张三",
    "phone": "13800000000",
    "role": "member"
  }
}
```

### 4.3 `GET /user/profile`

#### 前端期望响应

```json
{
  "userInfo": {
    "realName": "张三",
    "name": "张三",
    "phone": "13800000000",
    "role": "admin"
  },
  "scoreSummary": {
    "volunteerPoints": 12,
    "honorPoints": 16
  }
}
```

#### 说明

- `userInfo.role` 是前端管理员判断核心字段
- `scoreSummary` 用于首页/我的页/趋势图展示

### 4.4 `GET /admin/users`

#### 前端请求参数

```json
{
  "page": 1,
  "pageSize": 50,
  "keyword": "张三"
}
```

#### 前端期望响应

```json
{
  "list": [
    {
      "_id": "user_id",
      "realName": "张三",
      "phone": "13800000000",
      "role": "member",
      "status": "active",
      "isDeleted": false
    }
  ],
  "total": 100
}
```

#### 字段说明

- `_id`：用户唯一标识，后续设置角色/禁用都依赖该字段
- `realName`：页面展示用户名
- `phone`：页面展示手机号
- `role`：`super-admin/admin/member`
- `status`：当前账号状态，可选 `active/disabled`
- `isDeleted`：前端也兼容使用该字段判断是否已禁用

#### 前端当前判断逻辑

- 可用账号：
- `!item.isDeleted && item.status !== 'disabled'`
- 已禁用账号：
- `item.isDeleted || item.status === 'disabled'`

### 4.5 `POST /admin/users/role`

#### 前端请求参数

设置管理员：

```json
{
  "targetUserId": "user_id",
  "targetRole": "admin"
}
```

回收管理员：

```json

  "targetUserId": "user_id",
  "targetRole": "member"
```

- `targetRole = admin`：设为管理员

## 6. 图表年份逻辑说明

图表组件目前没有接真实逐年统计接口。

当前前端逻辑：

- 以当前年份为基准，生成最近 5 年
- 当前年显示 `volunteerPoints + honorPoints`
- 其余年份默认显示 `0`

这只是前端占位展示逻辑，不代表后端年度统计结果。  
如果后端后续提供“按年积分趋势”接口，前端可直接切换为真实数据。

## 7. 联调注意事项

### 7.1 角色字段必须统一

后端请确保所有涉及用户信息的接口，角色字段统一返回：

```json
"userInfo": {
  "role": "member"
}
```

不要混用：

- `isAdmin`
- `adminRole`
- `permission`
- `userType`

前端当前核心依赖是 `role`。

### 7.2 用户禁用状态建议统一

建议后端统一返回以下之一，最好两者都返回：

```json
{
  "status": "disabled",
  "isDeleted": true
}
```

这样前端兼容性最好。

### 7.3 用户列表需支持分页

`GET /admin/users` 当前前端会自动分页拉全量，因此后端请保证：

- `page`
- `pageSize`
- `list`
- `total`

这些字段行为稳定。

### 7.4 `super-admin` 不允许被普通管理员操作

前端已经做了限制，但后端仍需做强校验，避免绕过前端直接调用接口。

### 7.5 导出页真实用户筛选尚未完全落地

当前前端页面样式已经完成，但“用户筛选”仍是占位数据。  
后续建议联调时新增一个真实用户搜索/筛选接口，或者让 `GET /admin/users` 支持用于导出页的远程搜索。

## 9. 结论

本次提交对后端最核心的新增对接点是：

- 新角色体系：`super-admin / admin / member`
- 新用户管理接口：
- `GET /admin/users`
- `POST /admin/users/role`
- `POST /admin/users/disable`
- 登录与用户资料接口中必须稳定返回 `userInfo.role`
- 用户列表中必须稳定返回 `_id / realName / phone / role / status / isDeleted`

如后端按本文档结构返回，当前前端即可完成管理员角色管理的基础联调。
